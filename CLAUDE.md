# RION 개발 가이드 (Claude 참조용)

## 1. 프로젝트 개요

한국 주식시장 특화 인텔리전스 서비스.
오늘의 테마/섹터 흐름을 빠르게 파악할 수 있는 의사결정 지원 도구.

자세한 내용 → `docs/product.md`, `docs/disign.md` 참조

---

## 2. 기술 스택

| 영역       | 기술                              |
| ---------- | --------------------------------- |
| 프레임워크 | Next.js (App Router) + TypeScript |
| 스타일     | Tailwind CSS                      |
| 서버 상태  | TanStack Query                    |
| ORM        | Prisma                            |
| DB         | Supabase (PostgreSQL)             |
| 배포       | Vercel                            |

---

## 3. 현재 개발 단계 (MVP)

**포함:**

- 버튼 클릭 → 네이버 뉴스 크롤링 → DB 저장 → 화면 표시
- Next.js Route Handlers로 API 처리

**제외 (나중에 추가):**

- LLM 요약/분석
- 자동 크론 스케줄링
- 실시간 알림
- 실시간 시세 연동

## 4. 협업 방식

**세션 시작 규칙:**

- 새 대화 시작 시 docs/logs/logs.md와 최신 session.md를 읽고 현재 상태를 파악한다.
- 오늘 진행할 plan을 사용자에게 먼저 말한다.
- 해당 plan.md를 검토하라고 사용자에게 요청한다.
- 사용자 확인 후 작업을 시작한다.

**역할 분리:**

- 사용자: 설계, 판단, 우선순위 결정
- Claude: 지시받은 코딩만 실행

**코딩 원칙:**

- 요청한 범위만 구현한다. 그 이상 하지 않는다.
- 요청하지 않은 기능, 리팩토링, 최적화, 에러 핸들링 추가 금지
- 요청하지 않은 주석, 타입, 유틸 함수 추가 금지
- "이런 것도 하면 좋을 것 같아서" 식의 판단 금지

**로그 규칙:**

- plan 완료 후 다음 plan으로 넘어가기 전, 반드시 로그 업데이트 여부를 먼저 확인한다.
- 확인 없이 다음 plan을 시작하지 않는다.
- plan 완료 시 반드시 `.claude/plan-active` 파일을 생성한다. (`touch .claude/plan-active`)

**보고서 규칙:**

- plan 완료 후 반드시 `docs/plans/{plan번호}/report.md` 를 생성한다.
- 보고서는 코딩 초보도 이해할 수 있도록 상세하게 작성한다.
- 어떤 파일을 만들었는지, 왜 만들었는지, 각각 무슨 역할인지 설명한다.
- 보고서 작성 전까지 plan이 완료된 것으로 간주하지 않는다.

**구현 단계 규칙:**

- plan에 정의된 모든 항목을 빠짐없이 구현한다.
- 각 작업/단계 완료 시: report.md 작성 → 로그 업데이트 확인 순서를 반드시 따른다.
- 모든 작업이 완료될 때까지 멈추지 않는다.
- `any`, `unknown` 타입 사용 금지.
- 작업 중 지속적으로 typecheck(`npx tsc --noEmit`)를 실행해 새로운 타입 에러를 만들지 않는다.
- 계획한 것만 한다. 계획에 없는 것은 하지 않는다.
- 모든 단계가 완료되면 사용자에게 완료 여부를 확인한다.
- 사용자가 완료를 확인하면 한글로 커밋 메세지를 제안한다.

**완료 검증 규칙:**

- 모든 plan 완료 시: `npx tsc --noEmit` + `npm run build` 실행
- UI 작업이 포함된 plan (Phase 2 이상): 위 두 가지 + `localhost:3000` 화면 확인 요청
- 화면 확인 시 스크린샷을 `docs/plans/{plan번호}/screenshot.png`로 저장하고 report.md에 첨부
- 화면 확인은 사용자가 직접 보고 OK 해야 완료로 간주
- 빌드 에러 발생 시 반드시 수정 후 다음 단계로 넘어간다.

**작업 중 문제 발생 시 규칙:**

- 예상치 못한 문제가 생기면 즉시 행동하지 않는다.
- 반드시 다음 순서로 설명한다:
  1. 어떤 문제가 발생했는지 (무슨 에러/충돌인지)
  2. 왜 이 문제가 생겼는지 (원인)
  3. 해결 방법 선택지 제시 (A안, B안)
  4. 사용자 확인 후 진행
- 사용자 확인 없이 임의로 방향을 바꾸지 않는다.

## 6. 파일 구조 (예정)

```
/app
  /(routes)           - 페이지 라우트 (프론트)
  /api                - API 엔드포인트 (백엔드)

/features             - 기능 단위 모듈 (피처 베이스)
  /themes             - 오늘의 테마 목록
  /theme-detail       - 테마 상세
  /stock-detail       - 종목 상세
  /alerts             - 알림 (나중)
  각 피처 내부:
    /components       - UI 컴포넌트
    /hooks            - TanStack Query 훅

/server               - 서버 전용 로직 (백엔드)
  /crawlers           - 크롤링
  /queries            - DB 쿼리

/prisma               - schema.prisma
/lib                  - 공용 유틸 (양쪽 사용)
```
